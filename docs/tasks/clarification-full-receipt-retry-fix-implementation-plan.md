# Clarification full-receipt retry-count & wrapped-native contract fixes — implementation plan

## Origin

Discovered during the `structure-refactoring` financial-parity audit (see `results/structure-refactor-drift-blockers.md`, `results/structure-refactor-drift-accounting-failure-analysis.md`, `results/structure-refactor-drift-required-changes.md`). Finding 1 and the original Finding 2 (MANTLE) are **pre-existing** defects, present byte-identically before the structural refactor — confirmed via `git show HEAD:<old-path>` diff. Finding 3 (POLYGON) was discovered as a companion instance of the same defect class during Phase 3 plan review (parallel `financial-logic-auditor`/`business-analyst`/`system-architect` review), after the user challenged the original Finding 2 fix and prompted on-chain re-verification. None are refactor regressions; fixing them is a prerequisite the user has requested before merging the refactor, to remove sources of cross-rebuild financial non-determinism.

## Scope

- **The code/config changes are global**, not session-scoped: `ClarificationPreparationHandler.java` and `network-descriptors.yml` apply to every session/universe in this deployment. `reset-derived.sh` renormalizes derived state for **all** sessions (`updateMany({})`, no universe filter), not just one. This is expected and safe (single-owner/household deployment per `docs/adr`), and it means other sessions with MANTLE/POLYGON wallets get the corrected wrap/unwrap detection automatically as an intended side effect — not scope creep.
- Verification is scoped to session/accountingUniverseId `df5e69cc-a0c0-4910-8b7d-74488fa266e2` (3 wallets) because it is the only universe with proven impact (Finding 1) / currently in production.
- No new networks, protocols, or asset families touched. Only MANTLE and POLYGON `wrapped-native` entries were independently re-verified against on-chain ABI/bytecode as part of this plan (see Finding 3); the remaining ~11 networks' `wrapped-native` entries were spot-checked and look canonical, but were not bytecode-verified — flagged as a follow-up audit item below, not blocking this fix.
- Backend only: `backend/core/src/main/java/com/walletradar/application/normalization/job/clarification/ClarificationPreparationHandler.java`, `backend/core/src/main/resources/network-descriptors.yml`

## Root cause (from audit)

### Finding 1 — clarification full-receipt retry count silently hard-coded to 1 (high value)

- `ClarificationPreparationHandler.fetchFullReceiptForClarificationOrMarkFailure(...)` is called by `MetadataClarificationWorkflowHandler.clarify(...)` with `properties.getMaxAttempts()` (`clarification.max-attempts: 3`), but internally passes the literal `1` to `clarificationFailureHandler.markMetadataFailure(...)` instead of forwarding its own `maxAttempts` parameter.
- `ClarificationPreparationHandler.fetchFullReceiptForReviewOrMarkFailure(...)` has the identical defect: called with `properties.getFullReceipt().getMaxAttempts()` (`clarification.full-receipt.max-attempts: 2`), but hard-codes `1` into `clarificationFailureHandler.markReceiptFailure(...)`.
- Effect: any single transient failure of a full-receipt explorer fetch (BlockScout/Etherscan-compatible) permanently exhausts retries after **one** attempt (`ClarificationPolicyService.nextFailureDecision`/`nextReceiptFailureDecision`: `nextAttempts >= Math.max(1, maxAttempts)` — with `maxAttempts` forced to `1`, exhaustion fires immediately), regardless of the configured budget.
- Proven live impact: BASE PancakeSwap-V3 LP-exit transaction `0x0a757aeeb58667c545017cd8e5cd60dc994a8945ed810c60ea2aed18688f4f7a` (tokenId 938761) is stuck `CONFIRMED`/`UNKNOWN` with `ROUTER_METHOD_OVERLOAD_UNSUPPORTED`, even though BlockScout returns the full decoded receipt right now — the LP position shows as open with $2067.71 phantom cost basis and BASE USDC realised PnL is understated by ~$16.82.
- **Post-implementation correction (Phase 6 re-audit, `financial-logic-auditor`)**: the LP-exit itself resolved exactly as predicted (LP-RECEIPT 938761 closes to 0/no basis). However, the ~$24.37 BASE USDC realised-PnL delta observed after the fix does **not** come from an LP-exit USDC leg (none existed on-chain — Blockscout confirms the exit was single-sided WETH-only). It comes from two separate Euler BASE lending transactions (`0x650f531a…` deposit, `0x9a9c0190…` withdraw) that were also stuck `UNKNOWN` from the same retry-count bug and are now correctly classified, recognizing ~$12.10 of accrued interest as income, which ripples into a later swap's realized PnL. Direction and materiality of the original estimate held; the causal mechanism was different. Verified via full pre/post `normalized_transactions` diff, not just the snapshot aggregate.

### Finding 2 — MANTLE `wrapped-native` points at the wrong contract entirely (latent, functional gap, not just a label)

- `network-descriptors.yml`: `networks.entries.MANTLE.wrapped-native = { contract: "0xdeaddeaddeaddeaddeaddeaddeaddeaddead1111", symbol: "WMNT" }`.
- On-chain verification (Mantlescan, verified source + ABI) shows `0xdead...1111` is the **`BVM_ETH` predeploy** (`OptimismMintableERC20`, constructor `("Ether", "WETH")`), inherited from the OP-stack fork prior to Mantle's gas-token migration from ETH to MNT. Its ABI exposes only `mint(address,uint256)`/`burn(address,uint256)`, callable exclusively by the `L2StandardBridge` — **it has no `deposit()`/`withdraw()` functions at all**. It represents bridged L1 ETH, unrelated to wrapping the native MNT gas token. `protocol-registry.json` already labels this exact address correctly: `"WETH (Mantle)"`, family `WRAPPER`, with note `"MNT is gas token. WETH = 0xdeaddeaddeaddeaddeaddeaddeaddeaddead1111"`.
- The real WMNT contract — the WETH9-style wrapper that actually implements `deposit() payable` (selector `0xd0e30db0`) / `withdraw(uint256)` (selector `0x2e1a7d4d`), i.e. exactly what `WrappedNativeSupport.WRAP_SELECTOR`/`UNWRAP_SELECTOR` are designed to detect — lives at a **different address**: `0x78c1b0C915c4FAA5FffA6CAbf0219DA63d7f4cb8` (verified via Mantlescan ABI: `Deposit`/`Withdrawal` events, `deposit`/`withdraw` functions, name `"Wrapped Mantle"`, symbol `"WMNT"`; also documented in `docs.mantle.xyz` governance/tokenomics pages).
- Correcting only the symbol (`WMNT` → `WETH`) at the *current* address, as first proposed, would have been wrong: it would relabel an inert address (one that can never receive a `deposit()`/`withdraw()` call, since it doesn't implement those functions) instead of fixing the actual functional gap. As currently configured, `hasWrappedNativeIdentity()` compares the tx's `toAddress`/transfer counterparties against `0xdead...1111`, so a real user wrapping/unwrapping native MNT via the genuine WMNT contract (`0x78c1b0...`) never matches — **synthetic WRAP/UNWRAP legs for MANTLE are silently never generated today**, not merely mislabeled.
- Not proven to have caused any of the 25 audit mismatches in the current dataset (no confirmed historical WRAP/UNWRAP tx found yet in the affected wallets' MANTLE history) — but this is a functional gap, not just a latent labeling risk: any past or future MNT wrap/unwrap by a tracked wallet is currently invisible to `WrappedNativeSupport`.
- Blast radius beyond `WrappedNativeSupport`: 6 call sites read `wrappedNativeContract(networkId)`/`wrappedNativeSymbol(networkId)` — `WrappedNativeClassifier`, `OneInchNativeSettlementSupport`, `NativeSettlementRecovery`, `HeuristicClassifier`, `NativeSettlementClarificationTrigger`, `BlockScoutNativeSettlementClarificationSupport`. All pair `nativeSymbol` (gas token) with `wrappedContract` for the same purpose (detect gas-token-wrapper evidence); there is no split-brain consumer that actually wants the old dead address, so the single-field config fix is correct for all of them simultaneously.
- `protocol-registry.json` gap (accepted, not blocking): the corrected MANTLE WMNT address `0x78c1b0c915c4faa5fffa6cabf0219da63d7f4cb8` has no entry in `protocol-registry.json` yet, unlike every other network's wrapped-native contract (ETHEREUM, ARBITRUM, OP-stack, etc., which all have a matching `family: WRAPPER`/`role: WRAPPER_TOKEN` entry). `WrappedNativeSupport`'s leg synthesis reads only `network-descriptors.yml`, never `protocol-registry.json`, so this does **not** affect financial correctness (legs/AVCO/replay are unaffected). The only downstream consumer of `protocol-registry.json` in this area is `ProtocolNameResolutionService`, which attaches a display-only `protocolName` attribution to WRAP/UNWRAP rows — until a follow-up registry entry is added, real MANTLE (and POLYGON, see Finding 3) wrap/unwrap rows will classify and carry correct legs/basis, but will show `protocolName = null` in the UI. This is explicitly accepted as out of scope for this fix (cosmetic attribution gap only) and tracked as a follow-up.
- **Observable/product-level "done" statement**: today, any native MNT wrap/unwrap transaction on a tracked wallet is invisible to the pipeline (no WRAP/UNWRAP transaction type, no synthetic legs, whatever raw transfer legs exist are classified by other heuristics or left `UNKNOWN`/misclassified). After this fix, such a transaction will be classified as `WRAP`/`UNWRAP` with AVCO-neutral synthetic legs against native MNT — the same behavior already working correctly for ETHEREUM/ARBITRUM/OPTIMISM/BASE etc.

### Finding 3 — POLYGON `wrapped-native` has the identical defect class (companion fix, folded in during the same rebuild)

- `network-descriptors.yml`: `networks.entries.POLYGON.wrapped-native = { contract: "0x7ceb23fd6bc0add59e62ac25578270cff1b9f619", symbol: "WETH" }`.
- On-chain bytecode verification (`eth_getCode` via Polygon RPC) confirms `0x7ceb23fd...` is the PoS-bridged WETH representation (Ethereum-origin token bridged to Polygon): **no `0xd0e30db0` (`deposit()`) selector** in its dispatch table. Its `withdraw(uint256)` is the ChildERC20 bridge-exit burn function, not a native-MATIC unwrap. `protocol-registry.json` already independently labels this exact address `"WETH"` (Canonical, family `WRAPPER`) — same pattern as MANTLE's dead address.
- The real WMATIC contract — WETH9-style, confirmed via bytecode to have **both** `0xd0e30db0` (`deposit()`) and `0x2e1a7d4d` (`withdraw(uint256)`) — is at `0x0d500b1d8e8ef31e21c99d1db9a6444d3adf1270`.
- Effect: identical to the pre-fix MANTLE bug — native MATIC/POL wrap/unwrap detection, and all 6 downstream classification/native-settlement-recovery call sites, are silently inert for POLYGON today.
- Folded into this plan (rather than deferred) because: (a) it is the exact same one-line YAML pattern as Finding 2, (b) this plan already forces a full `--skip-frontend` renormalization + auditor rerun, so fixing it now costs nothing extra, and (c) deferring it would require a second full-reset cycle later for the same bug class. POLYGON is a materially more heavily-used network than MANTLE in this deployment's tracked wallets, raising the odds of real historical impact.
- Same `protocol-registry.json` gap applies (no entry yet for `0x0d500b1d8e8ef31e21c99d1db9a6444d3adf1270`) — accepted for the same reason as MANTLE (cosmetic `protocolName` attribution only).

## Changes (upstream-first)

1. **`ClarificationPreparationHandler.fetchFullReceiptForClarificationOrMarkFailure`** — replace the literal `1` passed to `markMetadataFailure(...)` with the method's own `maxAttempts` parameter.
2. **`ClarificationPreparationHandler.fetchFullReceiptForReviewOrMarkFailure`** — replace the literal `1` passed to `markReceiptFailure(...)` with the method's own `maxAttempts` parameter.
3. **`network-descriptors.yml`** — correct `MANTLE.wrapped-native.contract` from `0xdeaddeaddeaddeaddeaddeaddeaddeaddead1111` (BVM_ETH bridge predeploy, no deposit/withdraw capability) to `0x78c1b0c915c4faa5fffa6cabf0219da63d7f4cb8` (the real WMNT contract). `symbol` stays `WMNT` (correct for the real WMNT contract). No code change needed elsewhere since `NativeAssetSymbolResolver`/`WrappedNativeSupport` read this field directly. `protocol-registry.json`'s existing `"WETH (Mantle)"` entry for `0xdead...1111` is untouched — it remains a separate, correctly-labeled bridged-ETH asset, unrelated to wrapped-native.
4. **`network-descriptors.yml`** — correct `POLYGON.wrapped-native.contract` from `0x7ceb23fd6bc0add59e62ac25578270cff1b9f619` (PoS-bridged WETH, no deposit/withdraw capability) to `0x0d500b1d8e8ef31e21c99d1db9a6444d3adf1270` (the real WMATIC contract), and `symbol` from `WETH` to `WMATIC`. Same reasoning and blast radius as change 3. `protocol-registry.json`'s existing `"WETH"` entry for `0x7ceb23fd...` is untouched.
5. No other call site in `ClarificationPreparationHandler` has this defect (`loadRawOrMarkMetadataFailure`, `fetchMetadataReceiptOrMarkFailure`, `fetchFullReceiptOrMarkFailure` already forward `maxAttempts` correctly — confirmed by reading the full file).
6. **New unit test(s)** proving the corrected contracts actually trigger synthesis, not just a static config-value assertion: extend `OnChainClassifierTest`'s existing "wrapped-native deposit/withdraw selector becomes WRAP/UNWRAP" pattern (already covers the OP-stack predeploy and Arbitrum WETH) with a MANTLE fixture (tx to `0x78c1b0c9...`, selector `0xd0e30db0`/`0x2e1a7d4d`) and a POLYGON fixture (tx to `0x0d500b1d...`, same selectors), asserting `WRAP`/`UNWRAP` classification with correct synthetic legs against native MNT/MATIC respectively.

## Risks

- **Behavior change under real conditions**: forwarding the real `maxAttempts` means transient full-receipt failures now retry up to 3×/2× before giving up, instead of failing immediately, with the correctly-paired existing delays: `clarification.retry-delay-seconds: 120` for the `max-attempts: 3` path, and `clarification.full-receipt.retry-delay-seconds: 300` for the `max-attempts: 2` path (verified in `application.yml`). This is the *intended* configured behavior — no new risk, but rebuild timing for the clarification stage may take longer (retry delay applies only on actual failures, not to already-cached/available evidence).
- **Explorer rate-limit exposure during full rebuild**: forwarding real `maxAttempts` means every previously-fail-fast full-receipt fetch now issues up to 2–3× the outbound calls to public block explorers (BlockScout/Etherscan-compatible) during the mandated full-dataset renormalization. Given this deployment's free-tier/public-RPC cost constraints, this is a real (if likely small, given existing batch/lease throttling) increase in explorer call volume concentrated in one rebuild window — worth monitoring during the verification rebuild, not a blocker.
- **Full reset required**: per `prod-rebuild-workflow.mdc`, any `normalization`/`clarification` change requires `--skip-frontend` (full derived-state reset + renormalization), not `--backend-only`. This will re-run classification for the whole dataset, which is a heavier verification step but is exactly the guarantee the user asked for ("подтверждение финансовой корректности" before merge).
- **MANTLE and POLYGON wrapped-native contract fixes** have no proven current-data impact (no historical WRAP/UNWRAP tx confirmed affected yet) but correct a functional gap (WRAP/UNWRAP detection was silently inert for both networks), not just a label — explicitly called out in acceptance criteria in case a MANTLE or POLYGON WRAP/UNWRAP transaction is found or newly detected during the full re-normalization that behaves differently than before (previously: no synthetic legs at all; after: correctly enriched WRAP/UNWRAP legs against native MNT/MATIC).
- **`protocol-registry.json` attribution gap (accepted)**: real MANTLE/POLYGON wrap/unwrap rows will classify and carry correct legs/basis after this fix, but will show `protocolName = null` in the UI until a follow-up registry entry is added for the two newly-activated wrapped-native contracts. Cosmetic only — no financial-correctness impact (confirmed: `WrappedNativeSupport` never reads `protocol-registry.json`).
