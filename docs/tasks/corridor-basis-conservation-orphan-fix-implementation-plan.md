# Corridor basis conservation orphan fix (CB-1/CB-2/CB-3) — implementation plan

## Origin

Discovered while verifying `docs/tasks/clarification-full-receipt-retry-fix-implementation-plan.md`: a full `--skip-frontend` rebuild triggered `CorridorBasisConservationGuard` (`HARD_FAIL`, promoted for the `structure-refactoring` refactor) to throw `CorridorBasisConservationException: 6 orphaned released carry-out(s) totalling $196.496499823863583898719715583154260`, blocking replay. The guard was temporarily softened to `WARN` (`JAVA_TOOL_OPTIONS=-Dwalletradar.conservation.guards.hard-fail=false`) to let that unrelated fix's verification complete. `financial-logic-auditor` Phase 1 classification (`results/accounting-failure-analysis.md`, `results/required-changes.md`, `results/protocol-rule-pack.md` — ADDENDUM "`CorridorBasisConservationGuard` 6-orphan classification", 2026-07-06) confirmed this $196.50 orphan is **pre-existing** (byte-for-byte identical `normalized_transactions` before/after that fix) and **not** the ADR-043 §A7 "blocked, needs shared-matcher owner" defect its authors anticipated — all three root causes have narrow, safe, seam-local fixes.

**Revision note (post Phase-3 review):** this plan was reviewed in parallel by `financial-logic-auditor`, `business-analyst`, and `system-architect`; all three returned REQUEST CHANGES with concrete, mostly non-overlapping findings against the actual source code. This revision folds all of them in. Key corrections from v1: CB-1's causal narrative (`isCorridorLeg`) was wrong and the fix is now a single shared-filter change, not two method-local patches. CB-2's "unconditional overwrite" premise was found to already exist in code — the real fix target is the sibling-*selection* logic, to be found via a reproduction test first. CB-3's fix location (`persistStatusEvidence`) was factually wrong, needs a prerequisite `toAddress` model addition, needs a negative case for not-yet-backfilled destinations, needs counterparty attribution, and should live in its own dedicated service/pass rather than inline in the status-persistence method.

## Root cause (from audit, corrected per Phase-3 review)

### CB-1 — USDT $150.00: one-sided stream-mirror exclusion (linking)

- `BybitEarnPrincipalTransferPairer` correctly co-event-pairs the true EARN/UTA legs (`correlationId = bybit-earn-principal-v1:9ed98986…` on both), proven by Bybit's own independent `walletBalance` running-total field jumping to exactly `150.2` at the UTA credit.
- `BybitStreamAuthorityCollapser` (`collapseMirrors()`) independently re-excludes the UTA leg (`excludedFromAccounting=true`, `BYBIT_STREAM_MIRROR_TRANSACTION_LOG`).
- **Corrected causal mechanism** (per `financial-logic-auditor` review — the v1 plan's claim that the primary pass has a corridor-prefix guard the secondary passes lack was wrong): `isCorridorLeg` (`BybitStreamAuthorityCollapserSupport.java:29`, via `CorridorCorrelationKeyFactory.isCorridorKey`) only recognizes the **`BYBIT-CORRIDOR:`** on-chain prefix (RC-9/ADR-006) or a non-blank `txHash` — it does **not** check `bybit-earn-principal-v1:` at all, and all passes (`collapseMirrors()` primary pass, `demoteEventCountMirrors`, `demoteResidualMirrors`) already operate on the same pre-filtered `docs` list, so there is no primary-vs-secondary asymmetry. The real reason this bug rarely surfaces is the primary pass's tighter `mirrorSignature` grouping (60s bucket + subAccount + sign) — not a missing skip. The three passes also all write the identical exclusion-reason string, so the stored document alone cannot disambiguate which pass fired for this specific row.
- **Before coding:** recompute `mirrorSignature`/`walletSignSignature`/`broadSignature` against the actual orphaned document to confirm which pass produced the exclusion. If ambiguous, apply the fix uniformly (see Changes §1 below, which already does this by construction).

### CB-2 — USDC $30.19 + LDO $6.17: correlation-ownership collision (linking)

- Both are clean, exact-equal-quantity subscribe/redeem pairs (30.1928=30.1928, 39.96=39.96), well inside the shared matcher's tolerance.
- **Corrected root cause** (per `system-architect` review — the v1 plan's "add an unconditional overwrite to `applyEarnPrincipalPair`" premise is factually wrong): `BybitEarnPrincipalTransferPairer.applyEarnPrincipalPair` (lines ~380-402) **already** unconditionally calls `transaction.setCorrelationId(correlationId)` on both legs it pairs, with no guard against a pre-existing `bybit-it-pair-v1:`/`bybit-it-roundtrip-v1:` value. The actual defect is upstream of that method: the pairer's own sibling-*selection* logic (the co-event/hold-FIFO matching that decides which candidate row is "the" UTA partner for a given EARN leg) never selects the true UTA row as a sibling in the first place — most likely because a same-quantity decoy leg (e.g. a FUND-side row within the same `CO_EVENT_MAX_SKEW` window) is consumed as the match first by whatever `paired[]`-marking FIFO order the selection uses. `BybitInternalTransferPairer` (which runs first, per `BybitNormalizationService.processNextBatch`: `repairAll()` → `collapseMirrors()` → `pairEarnPrincipalTransfers()`) then claims that unclaimed UTA row's correlationId with its own `bybit-it-pair-v1:`/`bybit-it-roundtrip-v1:` id, and nothing ever revisits it.
- `ReplayPendingTransferKeyFactory.isBybitEarnPrincipalPairedTransfer` requires `correlationId.startsWith("bybit-earn-principal-v1:")` to route a leg to the `corr-family:` queue (confirmed exact, `ReplayPendingTransferKeyFactory.java:39-64`); because the UTA leg's id doesn't match, it routes instead through `isBybitEarnInternalTransfer` to the shared same-UID FIFO queue (`bybit-earn-carry:<uid>:<asset>`) — a different, wrong queue. The `corr-family:` slot the EARN leg released is never dequeued. Orphan.
- **Investigation required before implementation:** reproduce the exact CB-2 fixture (an EARN leg, a same-quantity FUND-type decoy row, and the true UTA leg, all inside `CO_EVENT_MAX_SKEW`) as a unit test first, and trace which predicate in the pairer's candidate-selection path mis-selects the decoy or fails to consider the true UTA row. Fix that selection defect. Once the right row is selected, `applyEarnPrincipalPair`'s existing unconditional `setCorrelationId` call already does the right thing — no change needed there.
- **Collateral-effect check:** if a fix causes the pairer to re-claim a UTA row previously stamped by `BybitInternalTransferPairer`'s two-sided `bybit-it-pair-v1:`/`bybit-it-roundtrip-v1:` id, the *other* leg of that now-broken IT pairing is left holding a singleton correlation id with no partner. This is very likely self-healing (that leg becomes a re-pairing candidate on the next batch, per `BybitInternalTransferPairer`'s `loadSingletons()`-style candidate query), but must be verified, not assumed — see Acceptance.
- Ordering note (verified, no fix needed): `BybitInternalTransferPairer.repairAll()` always runs before `BybitEarnPrincipalTransferPairer.pairEarnPrincipalTransfers()` in the fixed, unconditional call order inside `BybitNormalizationService.processNextBatch`. This convergence is safe today ("last writer in a fixed pipeline wins," an existing pattern also used by `bybitInternalTransferExternalCpReclassifier` in the same file) but nothing currently enforces that order — add a structural test pinning it (see Acceptance) since the fix's correctness depends on it.

### CB-3 — 3× LI.FI bridge-settlement orphans: `BRIDGE_OUT` to an untracked third-party wallet (classification, genuine type-model gap)

- Three ARBITRUM/KATANA→BASE LI.FI sends (via Jumper.exchange aggregator) from the tracked "Metamask" wallet are classified `BRIDGE_OUT` awaiting a same-owner destination.
- LI.FI's own status API (`li.quest/v1/status?txHash=`) confirms all three are `status=DONE`/`substatus=COMPLETED` and land at `0xcd74f91e4d2a49903462d58d6951136a527a5dea` on Base — **not** one of this session's 4 tracked wallets. Repeated over a 3-month span: a real third-party payment, not a stuck/pending self-bridge.
- **Corrected fix location** (per `financial-logic-auditor` review — the v1 plan named the wrong method): `persistStatusEvidence` (`LiFiBridgePairLinkService.java:980-983`) only merges protocol status into raw evidence and saves it (`rawTransactionClarificationEnricher.mergeProtocolStatus(...)`) — it never touches `matchedCounterparty` or any `NormalizedTransaction`. The `matchedCounterparty = status.receivingTxHash()` assignment actually happens in `seedSourceAnchorFromStatus` (line ~1014), called from `seedSourceCounterparty` (line ~219) — the real fallback path reached when a `DONE` status has no destination document found. Additionally, `persistStatusEvidence` is called from **two** separate code paths (synchronous branch at line ~188, parallel-future branch at line ~248); this reinforces why the reclassification logic must not be embedded there (see Changes §3).
- **Prerequisite model gap** (confirmed by both `financial-logic-auditor` and `business-analyst`): `LiFiBridgeStatus` (record) has **no `toAddress` field today**, and `LiFiStatusGateway.parse()` never extracts it from the LI.FI response, even though LI.FI's `/v1/status` payload does return a top-level `toAddress` once `status=DONE` (confirmed against LI.FI's public API docs). This must be added before any tracked-wallet comparison is possible.
- **Layering issue** (per `system-architect` review): every other "reclassify a `NormalizedTransactionType` based on newly-proven counterparty/destination evidence" case in this codebase (`OwnWalletBridgeMistypeCorrectionService`, `RegistryBridgeInboundTypeCorrectionService`, `KnownBridgeRouterExternalTypeCorrectionService`, `MultiCounterpartyCorrectionService`, and the closest functional analog, `unmatchedBridgeInboundPricingFallback.orphanInbounds` — "demote orphan `BRIDGE_IN` to market-priced ACQUIRE") is implemented as its own dedicated, named `timedPass` in `LinkingBatchProcessor`, not inline inside a status-persistence method. CB-3 should follow the same shape.
- Classification-spec §6.4 (`BRIDGE-01`) only models "no matching `BRIDGE_IN` found yet" (temporary backfill-lag: `missingDataReasons += BRIDGE_DESTINATION_UNKNOWN`) — it has no terminal state for "destination confirmed by the bridge's own settlement API, and it's definitively not us." Genuine type-model gap, not merely a missing linking correlation; no new canonical type needed (`EXTERNAL_TRANSFER_OUT` already exists and already models third-party sends).
- Source vs. destination `amountUSD` in the LI.FI responses show near-zero price drift on these stablecoin/near-stablecoin legs (e.g. $3.1440 sent vs $3.1350 received) — fixing this realizes close to $0 net P&L; the value is attribution correctness/determinism, not recovering understated gains.

## Changes (upstream-first)

1. **`BybitStreamAuthorityCollapser`/`BybitStreamAuthorityCollapserSupport`** — extend the shared `isCorridorLeg` skip (or the initial candidate filter that all three demotion passes read from — `collapseMirrors()` lines ~93-96) to also treat any row whose `correlationId` starts with `BybitEarnPrincipalTransferPairer.EARN_PRINCIPAL_CORRELATION_PREFIX` (with `continuityCandidate=true`) as protected, i.e. excluded from the demotion candidate pool entirely. Fixing it once at the shared-filter level (rather than patching `demoteEventCountMirrors` and `demoteResidualMirrors` separately) protects **all** current and future downstream passes uniformly (including `unifyOpposingCorrelations` and the orphan-drift pass), and sidesteps the need to prove exactly which of the three passes produced this specific orphan.
2. **`BybitEarnPrincipalTransferPairer`** — first, write a reproduction unit test with an EARN leg + same-quantity decoy FUND-type leg + true UTA leg (all inside `CO_EVENT_MAX_SKEW`) to identify the exact predicate in the sibling-selection path that mis-selects or fails to select the true UTA row. Fix that selection defect so the true UTA row is chosen as the sibling (at which point `applyEarnPrincipalPair`'s existing unconditional `setCorrelationId` call already produces the correct shared correlation — no separate "add an overwrite" change is needed, confirming/superseding the v1 plan's premise). Re-run for both already-affected rows on the next incremental batch/full-reset normalization pass.
3. **CB-3, in three ordered steps:**
   a. **Model prerequisite:** add a `toAddress` field to `LiFiBridgeStatus`, parse it in `LiFiStatusGateway` from the LI.FI `/v1/status` response, and update `toDocument()`/`fromDocument()` accordingly. Because `scripts/reset-derived.sh` (invoked by `--skip-frontend`) does `$unset: clarificationEvidence` on all `raw_transactions`, a full rebuild will force a fresh LI.FI status fetch that populates this new field — no separate backfill/reprocessing step is needed for the verification rebuild, but note this explicitly so it isn't mistaken for a gap.
   b. **New dedicated service** (e.g. `LiFiForeignDestinationReclassificationService`, package `application.linking.pipeline.clarification`), modeled on `OwnWalletBridgeMistypeCorrectionService`'s shape (constructor-injecting `AccountingUniverseService` + `SessionWalletAdjacencyService`, reusing the same tracked-wallet membership pattern rather than inventing a new accessor):
      - Query `NormalizedTransaction` rows where `type=BRIDGE_OUT`, protocol=LI.FI, `continuityCandidate=false`, with persisted LI.FI status evidence `status=DONE` **and** `substatus=COMPLETED` (not merely `DONE` — a `DONE`+`PARTIAL` settlement is a materially different, out-of-scope case) and a non-null `toAddress`.
      - **Negative case (from `results/protocol-rule-pack.md` RULE-PACK-4 item 5, dropped from v1 — must be restored):** do NOT reclassify while the destination network is not yet fully backfilled for the session, and do NOT reclassify while status is `PENDING`/`FAILED` — only a confirmed `DONE`+`COMPLETED` with a resolved `toAddress` is authoritative.
      - Compare `toAddress` (case-insensitive) against the session's tracked-wallet set via `accountingUniverseService.shareUniverseMembers(...)` / `sessionWalletAdjacencyService` — the same mechanism `OwnWalletBridgeMistypeCorrectionService` already uses for the structurally identical "is this address ours" question (do not introduce a second, inconsistent tracked-wallet accessor).
      - On a confirmed non-match: reclassify the source transaction's principal flow from `BRIDGE_OUT`/TRANSFER to `EXTERNAL_TRANSFER_OUT` with a SELL/DISPOSE role, valued at market-at-timestamp. Set `counterpartyAddress = toAddress` with confidence `EXACT` (per RULE-PACK-4 items 21-22 — proven by the bridge's own settlement API, not inferred). Clear the `BRIDGE_DESTINATION_UNKNOWN` missing-data reason, mirroring `OwnWalletBridgeMistypeCorrectionService.removeMissingReason`.
      - Idempotent by construction: the candidate query only matches rows still `BRIDGE_OUT`, so re-running over an already-reclassified row is a no-op.
   c. **Register the pass:** add `timedPass("liFiForeignDestinationReclassification", ...)` in `LinkingBatchProcessor`, placed immediately after `timedPass("liFiBridgePairLink", ...)` (line ~137) so status evidence is already persisted — the same slot pattern `unmatchedBridgeInboundPricingFallback.orphanInbounds` occupies for the inbound-side terminal reclassification.
   d. **Docs:** add the addendum to `docs/pipeline/normalization/classification-spec.md` §6.4 (`BRIDGE-01`), stated precisely as: "when a LI.FI/Jumper `BRIDGE_OUT`'s status resolves to `status=DONE` **and** `substatus=COMPLETED` with a `toAddress` that does not match any tracked wallet in the active accounting universe, and the destination network is fully backfilled for the session, reclassify to `EXTERNAL_TRANSFER_OUT`/SELL with `counterpartyAddress=toAddress` (confidence EXACT); do not apply while status is not yet `DONE`+`COMPLETED` or while the destination network's backfill is incomplete."
4. **Scope guard (all three):** none of these changes touch `ReplayPendingTransferMatcher`/`findUniqueCompatibleQueueIndex` (the ADR-043 §A7-protected shared matcher), the `bridgeSettlementInboundCarry` pass-through reserves, or the RC-9 same-owner ETH corridor. A synthetic same-owner LI.FI bridge (`toAddress` == a tracked wallet) must continue to route through the existing pass-through corridor unaffected.

## Risks

- **CB-1/CB-2 interaction risk:** both touch Bybit linking passes that run in the same batch (`BybitNormalizationService.processNextBatch`); implement and test them independently (separate commits, per Scope) but verify via full rebuild that they don't interact unexpectedly.
- **CB-2 is an investigation-first fix, not a known one-line change:** the plan intentionally does not prescribe the exact code fix for the sibling-selection defect because it hasn't been located yet (v1's proposed fix was confirmed to already exist in code and therefore a no-op). Budget time for the reproduction-test-first step in Changes §2; do not let an implementer skip straight to "add an overwrite" again.
- **CB-3 blast radius:** the reclassification only fires on confirmed `DONE`+`COMPLETED` + definitively-foreign `toAddress` + fully-backfilled destination network — must not affect the RC-9 same-owner ETH corridor, any bridge still `PENDING`/awaiting backfill, or any not-yet-backfilled destination network. Covered by the explicit regression tests in Acceptance item 1.
- **CB-3 requires a model/schema change before the behavioral fix is even possible** (`toAddress` doesn't exist in `LiFiBridgeStatus` today) — sequence this as step 3a before 3b/3c, not in parallel.
- **Full reset required:** per `prod-rebuild-workflow.mdc`, linking/classification changes require `--skip-frontend`, not `--backend-only`.
- **Guard restoration is a separate step, not automatic:** this plan's acceptance only confirms the 6 known orphans clear on this dataset, checked installation-wide. Do not assume zero orphans generalizes to all future data — restore `HARD_FAIL` only after the confirming rebuild, and keep monitoring for new breach classes since `HARD_FAIL` will again block replay on any recurrence.
- **Low priority, not urgent:** per the auditor's own recommendation, this is safe to bundle into a normal work cycle rather than rush — the $196.50 total is immaterial and the guard is already `WARN`, not currently blocking anything in its softened state.
